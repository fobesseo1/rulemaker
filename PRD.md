# 📄 제품 요구사항 정의서 (PRD)

**프로젝트명:** 마불(Marbull) 경량 제너러티브 코인 관리자 플랫폼 및 API

---

## 1. 프로젝트 개요 (Overview)
본 프로젝트는 앱 내 사용자 보상으로 지급되는 '제너러티브 코인(캐릭터 기반 다중 레이어 조합 이미지)'의 구성 요소(파츠)와 확률을 관리자가 손쉽게 통제하고 배포할 수 있는 **어드민 웹 UI 및 무상태(Stateless) API** 구축을 목표로 합니다.

## 2. 핵심 아키텍처 원칙 (Core Architecture Principles)
본 시스템은 백엔드 리소스를 최소화하고 클라이언트 자원을 극대화하는 아래의 핵심 원칙을 따릅니다.

1. **서버의 무상태성 (Stateless Backend):** 백엔드 API는 이미지 병합 작업을 수행하지 않습니다. 난수(SHA-256)를 발생시켜 조합된 파츠의 **JSON 메타데이터와 고유 해시값만 프론트엔드로 전달**합니다.
2. **클라이언트 렌더링 (Client-side Rendering):** 실제 파츠 이미지들을 겹쳐 하나의 코인으로 렌더링하는 작업은 사용자 앱(프론트엔드)에서 CSS(`position: absolute`, `z-index`)를 사용하여 처리합니다.
3. **브라우저 기반 이미지 최적화 (Zero-cost Converting):** 관리자가 고화질 투명 PNG를 업로드 시, 서버 자원을 쓰지 않고 **관리자 브라우저의 Canvas API를 활용해 초경량 WebP 포맷으로 자동 변환** 후 스토리지에 업로드합니다.

---

## 3. 기능 요구사항 (Functional Requirements) - 어드민 대시보드

### 3.1. 코인 컬렉션 관리 (Collection Management)
- **컬렉션 생성:** 새로운 코인 시리즈(예: "마불이 기본팩", "여름 바캉스팩")의 이름을 지정하여 프로젝트 단위로 신규 컬렉션을 생성합니다.
- **컬렉션 조회 및 수정:** 생성된 컬렉션 목록을 대시보드에서 갤러리 형태로 확인하고, 진입하여 상세 내역(파츠/확률 등)을 관리할 수 있습니다.

### 3.2. 레이어(Layer) 및 파츠(Parts) 자유 구성 시스템
포토샵과 유사한 직관적 요소를 활용하여 코인의 골조와 파츠를 구성합니다.
- **레이어 관리 설정:** 
  - 코인을 구성할 레이어를 무제한으로 추가/삭제하거나 이름을 변경할 수 있습니다. (예: Background, Body, Eyes)
  - **Z-index 부여:** 단위 레이어별로 Z-index 번호를 파라미터로 관리하며, 드래그 앤 드롭을 통해 직관적으로 우선순위를 변경할 수 있습니다. (숫자가 작을수록 하단 배치)
- **파츠 자산 관리:**
  - 특정 대상 레이어에 포함할 파츠 이미지들을 드래그 앤 드롭으로 다중 업로드합니다.
  - **이미지 규격 검증:** 파일 업로드 시 **512x512px 사이즈의 투명 배경 PNG**인지 클라이언트 단에서 검증하고, 미충족 시 업로드를 차단합니다.
  - **파츠 속성 입력:** 업로드된 각각의 파츠에 '파츠 이름(예: 선글라스)'과 시스템 내 '등장 확률(예: 5%)'을 관리자가 직접 입력합니다.

### 3.3. 수동 조합 및 시각적 테스트 캔버스 (WYSIWYG 렌더링)
- **실시간 캔버스 (Preview):** 화면 중앙에 512x512px 크기의 작업 미리보기용 실시간 캔버스를 제공합니다.
- **수동 조합 테스트:** 분류된 레이어들의 특정 파츠 항목을 에디터상에서 클릭(선택) 시, 중앙 캔버스에 즉시 Z-index 순서대로 겹쳐서 적용되어 오작동 유무(레이어 꼬임 등)를 시각적으로 검수할 수 있습니다.
- **확률 계산 모니터링:** 수동으로 조합 중인 결과물들의 '최종 발현 확률'(선택된 파츠들의 곱연산 등을 취합)을 화면 캔버스 하단에 실시간 수치로 표기합니다.

### 3.4. SHA-256 기반 랜덤 가챠 시뮬레이터 (Simulator)
수동 조합 외에도, 실제 확률에 근거한 난수 동작의 무결성을 로컬에서 돌려보는 시뮬레이터를 제공합니다.
- **로컬 난수 시뮬레이션:** [랜덤 버튼] 클릭 시, 시스템 내 입력된 데이터 기반으로 무작위 확률 시뮬레이션을 가동하여 랜덤 조합물을 도출합니다.
- **로직 정합성 메커니즘:** 로컬 시뮬레이션에서도 프로덕션 백엔드 API와 동일하게 사용될 **SHA-256 기반 논리적 난수 생성 로직**을 적용하여 정확도를 검증합니다.

### 3.5. 프론트엔드 최적화 배포 파이프라인 (Deploy Pipeline)
- **로컬 무손실 WebP 변환:** 배포 프로세스 진입 시, 관리자의 브라우저 백그라운드에서 Canvas API를 활용해 등록된 원본 PNG 파일 전체를 용량이 가벼운 Lossless WebP 체계로 자동 리렌더링 및 변환합니다.
- **Supabase(BaaS) 동기화:**
  - 변환 처리된 WebP 스토리지 이미지 배포본 일괄 업로드.
  - Supabase PostgreSQL Database에는 최종 처리된 파츠 이름, WebP Storage 접근 상대/절대 URL 및 부여된 확률값이 내장된 통합 **JSON 룰셋(메타데이터)**을 저장합니다.
- **배포망 API 동작 점검:** 배포 직후 관리자 콘솔단에서 `POST /api/generate-coin`을 직접 Ping 테스트 발송하여 반환되는 JSON 응답 형태를 직접 디버깅할 수 있는 텍스트 뷰어를 제공합니다.

## 4. 백엔드 API 구동 요구사항 (API Specifications)

앱 내에서의 특정 조건 달성 시 백엔드로 호출되는 가벼운 무상태 API의 명세 요구사항입니다.

### 4.1. 동작 엔드포인트 명세
- **Method & Endpoint:** `POST /api/generate-coin`
- **Request Payload Example:** 
  ```json
  {
    "userId": "user_id_1234",
    "collectionId": "base_pack_01"
  }
  ```

### 4.2. 결정론적 난수 및 유전자 혼합 로직 (SHA-256)
코인의 고유성을 영구히 유지하고 외부 예측성을 차단하기 위한 해시화 매커니즘 방식을 정의합니다.
1. **기반 시드(Seed) 추출:** `userId` 값과 개별 요청의 `Timestamp`, 그리고 시스템 내장 `Secret Key` 등을 혼합시킨 원형 텍스트 시드값을 1차로 생성합니다.
2. **단방향 해싱 (Hashing):** 시드값을 **SHA-256 알고리즘** 기반 해싱하여 총 64자리의 16진수 문자열 해시 식별자(CoinHash)를 부여받습니다.
3. **확률/파츠 매핑 및 바이트 분할 (Trait Allocation):**
   - SHA-256 해시를 분할(Sub-string)합니다. (예: 인덱스 0~3은 Z-index 1 레이어용 값으로 할당).
   - 16진수로 잘려진 값을 다시 10진수로 치환하여 난수로 선언합니다.
   - Database에 배포되어 있는 해당 `collectionId` 기준 파츠의 '확률 JSON 룰셋' 정보와 바이트 단위 매칭을 취하여 해당하는 파츠가 확정(낙점)되도록 구현합니다.

### 4.3. 결과 응답 (Response) 정책
레이어 속성을 포함시킨 순수 JSON 만을 반환합니다.
- **Response Payload Example:**
  ```json
  {
    "success": true,
    "coinHash": "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855",
    "collectionName": "마불이 기본팩",
    "traits": [
      {
        "layerName": "Background",
        "zIndex": 1,
        "partName": "노란색 배경",
        "imageUrl": "https://[SUPABASE...]/bg_yellow.webp"
      },
      {
        "layerName": "Body",
        "zIndex": 2,
        "partName": "마불이 기본 몸통",
        "imageUrl": "https://[SUPABASE...]/body_base.webp"
      },
      {
        "layerName": "Eyes",
        "zIndex": 3,
        "partName": "선글라스",
        "imageUrl": "https://[SUPABASE...]/eyes_sunglasses.webp"
      }
    ]
  }
  ```

## 5. 클라이언트(프론트엔드/앱) 렌더링 가이드라인 방침
해당 시스템을 연동하게 될 사용자 앱 부분에서의 작업 원칙 가이드입니다.
1. 앱 내부에서 발급 JSON 페이로드를 전달받으면 `traits` 목록을 조회하여 나열된 WebP 이미지 URL들을 백그라운드 API 자원(예: `Promise.all` 등)을 이용해 비동기로 모두 일괄 다운로드/캐싱 처리합니다.
2. 이미지들의 준비가 완료 시, 투명화된 WebP 이미지의 특성을 활용하여 CSS/HTML 레벨에서 뼈대가 될 레이어 속성(`position: absolute`) 및 응답 지시된 `zIndex` 순서에 맞도록 오버레이 처리하여 사용자에게 최종적으로 제너러티브 코인 결과를 단일 UI로 보여줍니다.
